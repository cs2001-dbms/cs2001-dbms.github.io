Lossless Join Decomposition & Dependency Preservation

Lossless Join Decomposition

For \(R\) decomposed into \(R_1, R_2\), we require that for every legal relation \(r\) on \(R\):

\[r = \pi_{R_1}(r) \bowtie \pi_{R_2}(r)\]

Sufficient condition: the decomposition is lossless if at least one of these is in \(F^+\):

\[R_1 \cap R_2 \rightarrow R_1 \quad \text{or} \quad R_1 \cap R_2 \rightarrow R_2\]

⚠️ This is sufficient, but only necessary when all constraints are FDs (other constraint types can change the picture).

Full check:

  • \(R_1 \cup R_2 = R\)
  • \(R_1 \cap R_2 \neq \phi\)
  • \(R_1 \cap R_2 \rightarrow R_1\) or \(R_1 \cap R_2 \rightarrow R_2\)

Example: Supplier-Parts (Lossy)

Supplier_Parts(S#, Sname, City, P#, Qty), with S# → Sname, City and (S#, P#) → Qty.

Bad decomposition: Supplier(S#, Sname, City, Qty), Parts(P#, Qty)

Common attribute is Qty — not a superkey of either side. Joining back produces extra tuples: lossy. It also fails to preserve (S#, P#) → Qty.

Example: Supplier-Parts (Lossless)

Good decomposition: Supplier(S#, Sname, City), Parts(S#, P#, Qty)

Common attribute is S#, a superkey of Supplier. Joining back reconstructs the original exactly: lossless, and all dependencies preserved.

Example: Two Decompositions, Different Outcomes

\(R = (A,B,C)\), \(F = \{A \rightarrow B,\ B \rightarrow C\}\)

Decomposition Lossless? Dependency Preserving?
\(R_1=(A,B),\ R_2=(B,C)\) Yes — \(R_1 \cap R_2 = \{B\}\), \(B \rightarrow BC\) Yes
\(R_1=(A,B),\ R_2=(A,C)\) Yes — \(R_1 \cap R_2 = \{A\}\), \(A \rightarrow AB\) No — \(B \rightarrow C\) can’t be checked without joining

Same relation, same FDs — the choice of split determines whether both properties hold.

Dependency Preservation

Let \(F_i\) be the set of dependencies of \(F^+\) to only the attributes in \(R_i\). A decomposition into \(R_1, ..., R_n\) is dependency preserving if:

\[(F_1 \cup F_2 \cup \cdots \cup F_n)^+ = F^+\]

If not, checking updates for FD violations requires an expensive join.

Worked Example (Cyclic FDs)

\(R(A,B,C,D)\), \(F = \{A \rightarrow B,\ B \rightarrow C,\ C \rightarrow D,\ D \rightarrow A\}\)

Decomposition: \(R_1(A,B),\ R_2(B,C),\ R_3(C,D)\)

A→B, B→C, C→D are trivially preserved (each lives entirely inside one \(R_i\)). The only one to check is D→A:

\[D \rightarrow C \ (R_3) \Rightarrow C \rightarrow B\ (R_2) \Rightarrow B \rightarrow A\ (R_1) \Rightarrow D \rightarrow A \text{ by transitivity}\]

All dependencies preserved.

Testing (Polynomial Method — Preferred)

result ← α
while (result changes) do
    for each Ri in the decomposition:
        t = (result ∩ Ri)+ ∩ Ri      -- closure computed w.r.t. F, restricted to Ri
        result ← result ∪ t
-- α → β is preserved iff β ⊆ result

Apply this test to every FD in \(F\) to confirm the whole decomposition preserves dependencies.

Worked Example (Polynomial Method)

\(R(A,B,C,D,E,F)\), \(F = \{A \rightarrow BCD,\ A \rightarrow EF,\ BC \rightarrow AD,\ BC \rightarrow E,\ BC \rightarrow F,\ B \rightarrow F,\ D \rightarrow E\}\)

Decomposition: \(R_1(A,B,C,D),\ R_2(B,F),\ R_3(D,E)\)

A→BCD, BC→AD (both fully inside \(R_1\)), B→F (inside \(R_2\)), D→E (inside \(R_3\)) are automatically preserved. Remaining to check: A→E, A→F, BC→E, BC→F.

Checking A → EF:

Round Apply Result grows to
Start — {A}
\(R_1\) \((A)^+ \cap R_1\) {A,B,C,D}
\(R_2\) \((ABCD)^+ \cap R_2\) {...,F} (via B→F)
\(R_3\) \((ABCDF)^+ \cap R_3\) {...,E} (via D→E)

Result contains E,F ⇒ A→EF preserved.

Checking BC → E, F: same process starting from {B,C} — reaches A,D via \(R_1\) (BC→AD), then F via \(R_2\) (B→F), then E via \(R_3\) (D→E). Both preserved.

All dependencies preserved — this decomposition is dependency preserving.

Superkey Calculation Formulas

Given a relation with \(N\) total attributes, the number of possible superkeys depends directly on the number of candidate keys and the size of those keys.

Case 1: Only 1 Candidate Key (Single Attribute)

If the relation has only one candidate key, and that key consists of a single attribute (e.g., Key = \(\{A_1\}\)): \[\text{Total Superkeys} = 2^{N-1}\]

Case 2: Only 1 Candidate Key (Composite Key)

If the relation has only one candidate key, and that key is a composite key formed by \(k\) attributes: \[\text{Total Superkeys} = 2^{N-k}\]

Case 3: Every Single Attribute is a Candidate Key

If the relation has \(N\) candidate keys (meaning every single attribute in the relation is its own independent candidate key): \[\text{Total Superkeys} = 2^N - 1\]

Case 4: Two Distinct Candidate Keys

When a relation contains two distinct candidate keys (which can be composed of one or many attributes), we must use the Principle of Inclusion-Exclusion to avoid double-counting the superkeys that contain both keys.

Generic Set Formula

\[\text{Max Superkeys} = (\text{Superkeys with } A_1) + (\text{Superkeys with } A_2) - (\text{Superkeys with both } A_1 \text{ and } A_2)\]

Mathematical Representation

Let \(m\) be the number of attributes forming the first candidate key, and \(n\) be the number of attributes forming the second candidate key. Assuming the two candidate keys share no overlapping attributes:

\[\text{Max Superkeys} = 2^{N-m} + 2^{N-n} - 2^{N-(m+n)}\]

📌 Note: If the two candidate keys share overlapping attributes, the exponent in the subtraction term changes from \((m+n)\) to the number of attributes in their absolute union: \(|\text{Key}_1 \cup \text{Key}_2|\).
